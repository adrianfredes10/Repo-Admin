Paso 1 de 12
🔐 Login + Autenticación
Lo primero que mostrás. Conecta todo el sistema de auth.
🎥 Qué mostrar en pantalla

  • Abrí el navegador en http://localhost:5174 sin estar logueado — debería redirigir a /login automáticamente.

  • Mostrá el formulario de login. Ingresá con admin@foodstore.com / admin12345.

  • Después de loguear, abrí DevTools → Application → Cookies → buscá la cookie access_token y mostrá que tiene el flag HttpOnly.

💬 Qué decir
"La autenticación usa cookies HttpOnly. Cuando el usuario hace login, el backend devuelve una cookie con el JWT. No está en localStorage — está en una cookie HttpOnly, lo que significa que JavaScript no puede accederla directamente. Esto previene ataques XSS que roben el token. Acá se puede ver en DevTools la cookie access_token con el flag HttpOnly activo."
📁 Archivos a tener abiertos en el IDE
📄src/api/authApi.ts— login, logout, me requests
📄src/stores/useAuthStore.ts— estado de auth con Zustand
📄src/components/InitApp.tsx— checkAuth al iniciar
📄src/pages/LoginPage.tsx— formulario
✅ Punto clave del enunciado: "Cookie HttpOnly" es un requisito explícito. Mostralo siempre, aunque dure 5 segundos.
⚠️ Asegurate de que el backend esté corriendo en :8000 antes de grabar. La cookie la setea el backend, no el frontend.

🛡️ Rutas Protegidas por Auth y Rol
Mostrás que no se puede entrar sin estar logueado, y que cada rol solo accede a lo suyo.
🎥 Qué mostrar en pantalla

  • Deslogueate y abrí la URL /categorias directamente en el browser → debe redirigir a /login.

  • Logueate como STOCK. Intentá entrar a /categorias → debe mostrar la pantalla 403 / Acceso denegado.

  • Mostrá en el IDE el componente ProtectedRouter.tsx — la lógica de redirección.

💬 Qué decir
"Tengo un componente `ProtectedRoute` que recibe los roles permitidos. Si el usuario no está autenticado, redirige a `/login`. Si está autenticado pero no tiene el rol necesario, redirige a `/403`. Pero esto es solo UX. La seguridad real está en el backend: cada endpoint tiene `Depends(require_role)`. Si alguien manipula el frontend, el backend igual devuelve 403."

📄src/router/AppRouter.tsx— árbol de rutas con ProtectedRoute anidadas

🗂️ Layout + Sidebar Dinámico
El sidebar muestra distintas opciones según el rol.
🎥 Qué mostrar en pantalla

  • Logueado como ADMIN: el sidebar muestra Dashboard, Categorías, Ingredientes, Usuarios, Productos, Pedidos.

  • Logueado como STOCK: el sidebar solo muestra Dashboard, Productos.

  • Logueado como PEDIDOS: el sidebar solo muestra Dashboard, Pedidos.

  • Mostrá en el sidebar el nombre del usuario y su rol abajo a la izquierda.

💬 Qué decir
"El sidebar no es estático. Usa `hasAnyRole` del store de Zustand para mostrar solo las rutas a las que el usuario tiene acceso. Un empleado STOCK solo ve Productos — no ve Categorías, Ingredientes ni Usuarios. Esto evita confusión y reduce superficie de ataque en el frontend."

📦 Productos — CRUD + Disponibilidad + Stock
La pantalla más compleja del Parcial 1. Ahora con roles diferenciados.
👑 Como ADMIN

  • Ver lista con todas las columnas: nombre, precio, stock, disponible, botones editar/borrar.

  • Crear un producto con ingredientes.

  • Toggle disponible/no disponible (el switch).

  • Editar stock inline (click en el número → input → Enter).

  • Click en el nombre → va a la página de detalle.

📦 Como STOCK

  • Misma lista PERO sin botones de crear/editar/borrar.

  • SÍ puede cambiar disponibilidad y editar stock.

  • Mostrá el mismo componente con distinto comportamiento según rol.

💬 Qué decir al mostrar la diferencia ADMIN vs STOCK
"La pantalla de Productos la ve tanto ADMIN como STOCK. Uso `hasAnyRole` del store para mostrar o esconder los botones de CRUD. Un empleado STOCK puede cambiar la disponibilidad y el stock — ese es su trabajo. Pero no puede crear ni borrar productos. El mismo componente, distintos permisos."
💬 Qué decir sobre la invalidación de cache
"Cuando cambio la disponibilidad, uso `setQueryData` en lugar de `invalidateQueries`. Esto actualiza la cache local inmediatamente, sin esperar un refetch. Es un patrón de optimistic update — la UI responde al instante. Si el servidor devuelve un error, podría hacer rollback, aunque en este caso confío en que el 200 es definitivo."

Paso 8 de 12
🔍 Detalle de Pedido + Cambio de Estado
La funcionalidad más importante del parcial: avanzar el estado del pedido.
🎥 Flujo a demostrar
1. Click en un pedido PENDIENTE
Mostrá el detalle: items con nombre y precio al momento de la compra, total, forma de pago, fecha.
2. Click "Confirmar"
El backend recibe POST /api/v1/pedidos/{id}/avanzar. El estado pasa a CONFIRMADO.
3. Click "Marcar en preparación"
Estado → EN_PREP. Los botones de acción cambian automáticamente.
4. Mostrá el Network tab
Cada avance es un POST /avanzar sin body. El backend aplica la máquina de estados.
💬 Qué decir sobre la máquina de estados
"Los botones que aparecen dependen del estado actual. Uso un objeto `ACCIONES` en el frontend que mapea cada estado a sus transiciones posibles. Pero esto es solo UX. Si intentás hacer una transición inválida (por ej: PENDIENTE → ENTREGADO), el backend devuelve 409 Conflict. La lógica de negocio está en el backend, el frontend solo muestra lo que tiene sentido."
📁 Archivos a mostrar
📄src/pages/PedidoDetallePage.tsx— ACCIONES map, botones de transición
📄src/hooks/usePedidos.ts— useAvanzarEstado, useCancelarPedido con invalidateQueries
✅ Esto cubre: "Pantalla cajero: cambia estados (Confirmado, en preparación, listo, entregado)" — requisito explícito del enunciado.

🔄 Modo Admin vs Empleado
El mismo componente, distintos permisos. Mostrarlo en vivo con dos logins.
🎥 Demo en vivo — secuencia sugerida

  • Con ADMIN logueado: ir a Productos → mostrá los botones "Nuevo Producto", "Editar", "Borrar", el toggle de disponible Y el campo de stock.

  • Cerrá sesión. Logueate como un usuario con rol STOCK.

  • Con STOCK logueado: ir a Productos → la misma URL, el mismo componente, PERO sin "Nuevo Producto", sin "Editar", sin "Borrar". Solo puede tocar disponibilidad y stock.

  • Mostrá el sidebar: STOCK solo ve Dashboard y Productos.

💬 Qué decir
"Esto es el modo Admin vs Empleado que pide el enunciado. El componente `ProductosPage` es el mismo para ADMIN y STOCK. La diferencia está en los renders condicionales que usan `hasAnyRole`. Un STOCK puede cambiar disponibilidad y stock porque eso es su función, pero no puede crear ni borrar productos. Todo esto es UX — el backend igual rechaza las operaciones no permitidas."

// En usePedidos — mostrar esto:
onSuccess: (_, id) => {
  qc.invalidateQueries({ queryKey: queryKeys.pedidos.list({}) });
  qc.invalidateQueries({ queryKey: queryKeys.pedidos.detail(id) });
  qc.invalidateQueries({ queryKey: queryKeys.pedidos.historial(id) });
}
💬 Decir
"Después de avanzar el estado, invalido tres caches: la lista de pedidos, el detalle de ese pedido, y el historial. TanStack Query automáticamente refetchea solo los datos que están siendo observados en ese momento."
